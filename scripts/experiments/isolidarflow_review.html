Architecture & Implementation Plan

The isolidarflow plan turns an isolated single-storey SLAM scan into a sharp-edged, modular 3D model (named walls/floor/ceiling/doors/windows/balcony-doors/columns/beams as a GLB) plus a 2D floorplan, with every dimension measured rather than assumed. It was executed as 10 sequential tasks, each with its own tests, committed individually.

Architecture Revision R1

Adopted mid-plan after reviewing early room-wise 3D previews in Blender, which showed two bugs: walls shared between two rooms were meshed once per room edge, producing duplicated/fragmented panels; and walls unclaimed by any closed room silently vanished from the model. R1's fix — walls own a feature graph; rooms own wall segments — replaced the original flat build_elements approach:

  • Feature: one detected relief element on a wall (kind ∈ face/groove/l_extrusion/beam_soffit/column_attach/opening), positioned in the wall's own 1D u-coordinate with its own z-range and signed offset from the reference face — so grooves, protrusions, and beam soffits are named and preserved, not flattened into an anonymous box.
  • WallSegment: the actual meshing unit — a wall clipped to one room's ownership range (or room_id=None for unclaimed stretches, so nothing is ever silently dropped), carrying only the features that fall inside it.
  • Ownership pass: every room-edge overlap per wall is collected first, then each wall splits once into segments at the union of room-boundary u-coordinates — eliminating the duplicate-meshing bug at the root instead of patching it after the fact.
  • Meshing stays CSG (manifold3d) behind one interface, segment_to_mesh: additive features union, subtractive features (grooves, openings) cut, watertightness asserted. Profile-sweeping was evaluated and deliberately deferred — features can vary in both u and z simultaneously (a door under a beam soffit), so a sweep-based mesher would need its own hand-built watertightness guarantees that manifold3d already provides.

Current status

All 10 tasks are implemented and committed (branch isolidarflow, 20 commits). The final whole-branch review verdict was “ready to merge with fixes” — no Critical issues; both top user priorities (accurate opening cutouts, preserved wall relief) verified working end-to-end on two real scans with sharp, watertight CSG geometry. Three Important issues were found and fixed:

  • Silent mesh-failure swallows in build_room_model — a failed per-segment mesh operation vanished with no trace, undermining R1's own “never silently drop a wall” guarantee. Fixed with a counted, identified drop-report (model.drops), surfaced in report.txt.
  • Room-closure vs. point-cap tradeoff undocumented — the CLI's default plane_max_points (1.5M, tuned for speed) doesn't reliably hit the plan's own ≥7-room acceptance bar; now called out explicitly in the README with the flag to override it.
  • An accidental 11.7 MB PDF committed during Task 9 — removed, .gitignore hardened.

CLI entry point

scripts/isolidarflow.py exposes run(in_path, out_dir, config=DEFAULT_CONFIG) and a matching CLI, orchestrating every stage from LAS load through GLB/DXF/SVG/manifest export. DEFAULT_CONFIG holds every threshold from all 10 tasks — RANSAC parameters, wall-grouping tolerances, furniture-removal distances, opening priors (7 ft door height, 2750 mm ceiling, balcony/window rules) — each with a one-line comment, no magic numbers inline. Config is fully overridable per call or via CLI flags, so tuning (e.g. raising the point cap for better room closure) never requires touching code.

Real-Scan Experimentation Journey

All experimentation ran against the real koushikexport.las SLAM scan of an occupied apartment. Each numbered phase names its script and output folder so results can be reproduced or re-inspected.

1 · Isolation

The unit-isolation stage correctly separated the scanned apartment from an adjacent neighbour building visible in the same point cloud, producing a clean 13.2 × 15.0 m footprint with the neighbour structure fully dropped. This became the trusted base cloud for every downstream experiment.

2 · 2D floorplan iteration (v1 → v3)

scripts/experiments/diag_floorplan2d_v1/v2/v3.py progressively fixed room-closure defects:

  • v1: 5 rooms detected — many real rooms merged or missing.
  • v2: added u-gap splitting (breaking wall runs at real gaps instead of bridging them) → 8 rooms.
  • v3: added midline recentring and T-junction endpoint-to-line snapping (closing wall joints v2 left dangling) → settled at 7 rooms, the most topologically correct of the three.

3 · Point-count sweep (counter-intuitive finding)

Re-running detection at increasing point budgets did not monotonically improve results. At ~9M points, room count inflated as wall-fitting began fragmenting single rooms into fake micro-rooms, while runtime scaled from 25s to 1238s. The sweet spot was ~3–6M points; beyond that, more data actively hurt topology while cost exploded.

4 · Gap-region investigation

A persistent “hole” in the reconstructed floorplan was suspected to be scan noise or bleed-through from the neighbour building. Height-colored density renders plus a raw density image proved it was neither — it was a real balcony with a railing, confirmed by the railing's distinct low-density signature and trajectory data showing the scan path walking along it.

5 · Sharp 3D solid previews (v1 → v5)

Iterated stepped-boolean wall solids with gps_time-driven walkthrough door cuts: 53 trajectory crossings → 12 door cuts. Later versions added hollow gaps under beams and split walls into per-room WallSegments, fixing a bug where shared walls were duplicated as separate solids per adjacent room (“one object per wall”).

6 · Opening-detection bug fixes

Duplicate door detections from repeated trajectory crossings on the same opening were merged, reducing a noisy 32 openings down to 14. Two oversized voids (11.67 m and 5.6 m wide) that were being misclassified as balcony_door were reclassified as low-confidence unknown_opening, preventing a wrong high-confidence label on structurally implausible gaps.

7 · Entrance/topology rebuild

Added exterior/interior wall classification via room-polygon-side probing and a room-adjacency graph, then used adjacency + size to predict which room was the balcony. An early version wrongly flagged the 45.2 m² living room as a balcony purely from adjacency; fixed by making room-size a mandatory (not optional) filter condition.

8 · Slice validation

Four height slices (floor, waist, mid, near-ceiling) showed walls as clean, high-confidence bright lines at waist and mid height with no floor-plane brightness confound. The near-ceiling slice unexpectedly revealed a solid-filled region, likely a dropped ceiling or beam over the balcony — validating height-aware detection as a real signal source, not just a diagnostic.

9 · Hough/threshold struggle

Five thresholding polarity attempts were tried on wall-line extraction (adaptiveThreshold in both polarities, Otsu in both polarities, Canny). Plain non-inverted Otsu was visually best but still imperfect — this remains an open weak point in the pure-CV wall-extraction path.

10 · Ground-truth comparison

Compared against the real architect floorplan: 117 m² including 4 balconies, 103.5 m² excluding them, versus the reconstruction's 111.7 m² across 8 rooms (vs. 15 real spaces). Overall footprint area matched well, but kitchen/living merged into one 45.2 m² blob (real: 32.65 + 11.36 m² separately), several bathroom partitions were under-detected, and zero balconies were reconstructed as enclosed rooms.

11 · Floorplan–ML fusion breakthrough

bbox_data.json (25 walls, 10 openings from an independent ML detection flow) was registered onto the LiDAR frame via a 180° rotation, achieving IoU = 0.896 with no free scale factor needed since both sources were already metric. Every one of the 25 walls was then confirmed against real LiDAR point support with measured (not assumed) dimensions — crucially recovering the kitchen/living divider that pure point-cloud detection had missed. A thickness-measurement bug (percentile-based, producing a falsely uniform ~0.3–0.37 m everywhere) was found and fixed via two-peak histogram detection, yielding realistic thickness values of 0.04–0.23 m.

Known open problems

  • The balcony railing still needs dedicated height-aware detection (2D offset-grid or Poisson-mesh-slice approach) — current methods under-detect it.
  • Balcony rooms are not yet closed into proper enclosed polygons in the reconstruction.
  • Some bathroom partition walls are still missing relative to the ground-truth floorplan.

The floorplan-fusion architecture

Rather than trying to force RANSAC/plane-detection thresholds to recover every thin wall, the fusion approach treats the user's independently-generated ML floorplan JSON (bbox_data.json) as the structural base — it already encodes complete topology (which walls exist, which rooms they border, where every door/window/balcony-door sits) — while the LiDAR point cloud remains the measurement source for true as-built dimensions, since construction always deviates from the architect's drawing.

Fusion works by registering the floorplan onto the LiDAR's axis-aligned frame. Because both sources are already in real metric units, no free scale parameter is needed — only a rigid transform (rotation × mirror × translation). The registration search tries the 4 candidate 90° rotations combined with an optional mirror flip, scoring each by IoU between the transformed floorplan's wall footprint and the LiDAR-derived room-union footprint. This converged cleanly: IoU = 0.896 at a 180° rotation with no mirror.

Once registered, each of the floorplan's 25 walls is no longer “discovered” — it's measured: a search band is placed at its known expected position in the LiDAR cloud, and the true endpoints (density half-max crossing), thickness (two-face peak detection in a perpendicular-offset histogram), and z-extent are read directly off the real points. Result: 25/25 walls were confirmed by real LiDAR point support, including the kitchen/living partition wall that the LiDAR-only pipeline had merged away entirely — directly closing the largest gap in the table above.

Proposed next refinement (not yet built)

The per-wall measurement step currently reads directly off raw, noisy LiDAR points via percentile/peak-histogram methods. The next planned improvement is to instead measure off Poisson-mesh cross-sections — the mesh already exists (mesh_isolated_v4_clean.obj) and its own smoothing/gap-filling behavior should finally close the one gap raw-point measurement couldn't: the balcony railing, whose real points are too sparse and fragmented for a clean line fit, but which Poisson likely bridges into one continuous surface at railing height.

Alternative approach found: HouseKraft/PointCloudTo3d

A public reference implementation (github.com/HouseKraft/PointCloudTo3d) solves the same problem end-to-end without a pre-existing floorplan: Poisson mesh reconstruction → slice the mesh's own cross-sections at just 3 heights (0.10m near-floor for door bottoms, 1.10m mid-wall, 2.00m upper-wall/window band) → rasterize each slice as a density image → run a trained ONNX object detector (RF-DETR), not classical thresholding, to find wall/window/balcony-door boxes → merge detections across the 3 heights by direction + perpendicular-offset similarity → query the full original point cloud at each merged wall's footprint for the true floor-to-ceiling Z range → export to IFC.

Their config's real numeric priors closely match what this session independently derived: ceiling height 2.7m, minimum door opening height 1.9m, minimum balcony-door width 1.2m — solid cross-validation from an unrelated codebase.

Notably, their classical-CV Hough-line path is explicitly labeled the “no-training baseline” fallback, not the primary method — direct validation that this session's Hough/Otsu/Canny threshold-tuning struggle (the “not great” koushik_rescue5 output) was fighting the wrong tool for the job. Their primary path also includes a “nano banana” step — Gemini 2.5 Flash Image performing image-to-image cleanup on each raster before detection, under a strict “zero hallucination” prompt (never add or omit structure, preserve real openings as gaps, no drift from the input framing/scale). A trained detector plus a constrained generative cleanup pass is a fundamentally more robust way to separate “real wall” from “noisy clutter” in a density raster than hand-tuned brightness thresholds.

Where these two approaches now converge

We don't need HouseKraft's discovery machinery — training an ONNX detector or calling an external VLM — because discovery is already solved via the user's own ML-detected floorplan, fused and LiDAR-confirmed at IoU 0.896 with all 25 walls verified. The one idea worth borrowing directly is narrower and cheaper: their measurement technique — slice the denoised, gap-filled Poisson mesh rather than raw noisy points — applied to our already-known wall positions, not to blind discovery. That's precisely the next refinement already scoped above.
